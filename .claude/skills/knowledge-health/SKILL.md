# Knowledge Health Skill

Run monthly (or before any major deal push) to see what's fresh, what's stale, and what's missing.

## Invocation

```
/knowledge-health                    # Full dashboard
/knowledge-health compete            # Filter to compete/ files only
/knowledge-health plays              # Filter to plays/ files only
/knowledge-health --gaps             # Only show gaps and missing items
```

---

## Step 1: Read Frontmatter from All Knowledge/ Files

Scan every `.md` file in `Knowledge/` (all subfolders, excluding `_competitor-template.md` and `README.md`).

For each file, extract from YAML frontmatter:
- `knowledge_type` (descriptive or procedural)
- `last_updated`
- `stale_after_days`
- `source`
- `owner`

Fallback if frontmatter is missing: look for a `*Last updated: [YYYY-MM-DD]*` line in the document body. If neither is found, mark `last_updated` as "Unknown".

Calculate: `days_old = today - last_updated`. Compare to `stale_after_days` threshold.

Status:
- 🟢 **Current**: `days_old < stale_after_days`
- 🟡 **Aging**: `days_old >= stale_after_days * 0.75` (within 25% of threshold)
- 🔴 **Overdue**: `days_old >= stale_after_days`
- ⬜ **Unknown**: No date found

---

## Step 2: Query CRM for Competitor Gaps

Query for competitors appearing in active or recently closed deals that have no battle card in `Knowledge/compete/`:

```soql
SELECT Competitor__c, COUNT(Id) deal_count, MAX(LastActivityDate) last_seen
FROM Opportunity
WHERE IsClosed = false
  AND Competitor__c != null
  AND LastActivityDate >= LAST_N_DAYS:90
GROUP BY Competitor__c
ORDER BY deal_count DESC
```

Also check closed deals from the last 6 months for competitor patterns in win/loss fields (`CompetitorLost__c`, `CloseReason__c`).

For each competitor found: check if a corresponding `Knowledge/compete/[Competitor].md` file exists. If not, flag as a gap.

---

## Step 3: Scan Win-Loss/ for Pending Retro Insights

Read all files in `Win-Loss/Wins/` and `Win-Loss/Losses/`. Look for the **"Knowledge Updates Triggered"** section (generated by `/retro`). Find any rows where `Status` = `Pending`.

Collect:
- The insight tag and description
- The source retro file name and date
- The target Knowledge/ file
- Days since the retro was written (how long it's been pending)

## Notes for Skills Using This Data

When other skills (call-prep, deal-health, pipeline-review) do inline freshness checks, they read the same frontmatter fields. The logic is consistent:
- Read `last_updated` and `stale_after_days` from frontmatter
- Calculate days old
- Flag if overdue (🔴) or aging (🟡)
- Omit the flag entirely if current (🟢) — don't clutter outputs that don't need it
